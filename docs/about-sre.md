# SLI / SLO / SLA / エラーバジェットと、その決め方・握り方について

## はじめに

SLI、SLO、SLA、エラーバジェットという語は、SRE や運用設計の文脈で頻繁に語られる。しかし、実務上の難しさは、これらの用語の定義を知ることそれ自体にはない。難しいのは、どの数値を意味のある観測として採用するのか、なぜその値を境界にするのか、そしてその境界をどうやって組織の行動ルールとして握るのか、という点にある。とりわけ SLO は、表面的には「目標値」として説明されがちだが、それだけでは不十分である。SLO が本当に機能するためには、それが組織の優先順位を切り替える境界として設計されていなければならない。

本稿ではまず、SLI、SLO、SLA、エラーバジェットとはそれぞれ何であり、互いにどう関係しているのかを整理する。そのうえで、SLO の本質を単なる数値設定ではなく「行動を切り替える境界」として捉え直し、さらに、その境界をどうやって開発側・運用側・ビジネス側の合意にまで持っていくのか、という問題を論じる。

## まず、SLI、SLO、SLA、エラーバジェットとは何か

最初に押さえるべきなのは、これら四つの語が同じものを言い換えているのではなく、サービス運用の異なる層を切り分けているということである。SLI は観測の概念であり、SLO はその観測に対して引く境界の概念であり、SLA はその境界の一部を外部契約にした概念であり、エラーバジェットはその境界を運用制御に変換する概念である。

SLI は Service Level Indicator の略であり、サービスの状態を表す指標である。ただし、ここでいう指標とは「取得できる数値一般」ではない。システムには CPU 使用率やメモリ使用量やキュー長や Pod 再起動回数など、さまざまな数値がある。しかし、取得できるからといって、それらすべてが SLI になるわけではない。SLI と呼ぶに値するのは、ユーザーが何かを達成できたか、あるいは体験が許容可能な品質で提供されたかを代理できる観測量である。たとえばログイン成功率、注文成功率、アップロード完了率、検索成功率、あるいは p95 レイテンシのようなものが候補になるのは、それらがユーザー体験の成否に比較的近いからである。逆に CPU 使用率は、システム内部の状態としては重要であっても、それ自体でユーザー体験の成功や失敗を直接表すわけではない。したがって CPU 使用率は補助指標や原因分析指標にはなっても、主要な SLI にはなりにくい。

SLO は Service Level Objective の略であり、その SLI に対して「どこまでを許容するか」を定めた境界である。たとえば「注文成功率 99.5%以上」「p95 レイテンシ 250ms 以下」といった形で表される。ここで重要なのは、SLO は理想や願望ではなく、運用上の線であるということだ。SLO は「できればこのくらい良くしたい」という目標ではなく、「この線を下回ったら通常運転とは見なさない」という判断基準である。したがって SLO の意味は、値そのものの美しさではなく、その値を境に何が通常で何が異常かを切り分けることにある。

SLA は Service Level Agreement であり、SLO を顧客との契約にしたものである。SLO が内部運用のための境界であるのに対して、SLA は外部に対する約束であり、違反時には補償やペナルティが発生しうる。ゆえに SLA は一般に SLO よりも緩く設定される。内部では早めに異常を検知し優先順位を切り替えたいが、外部契約の水準までその感度を上げると、組織は不必要に硬直しやすいからである。内部で守る線と外部に約束する線は、同じである必要はないし、むしろ分けて考えるほうが自然である。

エラーバジェットは、SLO の裏返しである。たとえば成功率 99.9% を SLO に置くなら、0.1% は失敗してよいことになる。この「失敗してよい余地」がエラーバジェットである。ここで重要なのは、エラーバジェットが単なる免罪符ではないということだ。これは「どこまで壊してよいか」を定義すると同時に、「どこまでなら攻めてよいか」を定義する。まだバジェットに余裕があるなら、ある程度積極的に変更を出せる。逆にバジェットを使い切っていれば、機能追加より安定化を優先するべきだという判断が導かれる。したがってエラーバジェットは、信頼性の議論を開発速度の議論と接続するための装置である。

以上をまとめると、SLI は観測、SLO は境界、SLA は契約、エラーバジェットは運用制御である。だが、この整理だけではまだ足りない。現場で本当に難しいのは、何を観測するか、どこに境界を引くか、そしてその境界をどうやって組織の行動に接続するか、だからである。

## SLO の本質は、良い数値を掲げることではなく、行動を切り替えることにある

SLO は「目標値」と説明されることが多いが、その理解は半分しか当たっていない。SLO の本質は、状態を表すことではなく、状態の変化に応じて組織の行動を切り替えることにある。より正確に言えば、SLO は「ある SLI がこの線を下回ったら、自分たちは通常運転をやめて別の優先順位に移る」という境界である。

この観点から見ると、SLO は常に「条件」と「行動」の組として理解されるべきである。たとえば「注文成功率が 99.5% を下回ったら、新規リリースを停止して原因調査を優先する」といった形である。このとき 99.5% という値は、単なる数値ではない。それは通常状態と対応状態を切り分ける境界である。逆に、「注文成功率 99.5%」とだけ書かれていて、その値を下回ったときに何をするのかが決まっていなければ、それはまだ十分に運用ルールになっていない。

したがって、SLO が意味を持つかどうかは、その値が厳密に正しいかどうかだけでは決まらない。その値をきっかけにして、実際に組織の優先順位が変わるかどうかによって決まる。観測だけしていても、悪化したときに誰も判断を変えないなら、それはダッシュボード上の情報にとどまる。SLO の価値は、観測能力ではなく、行動変化能力にある。

ここでいう行動とは、たとえばリリースを止めること、ロールバックすること、インシデント対応に移ること、機能開発より安定化を優先すること、重要導線だけを守るためにトラフィック制御を行うことなどである。あるいは、エラーバジェットの消費率に応じて、注意喚起、レビュー強化、リリース制限、リリース停止といった段階的な行動を定めることも含まれる。重要なのは、これらの行動が事後的な感情や政治で決まるのではなく、あらかじめ「この状態ならここまではやる」と握られていることである。さもなければ、境界を越えるたびに毎回人間同士の調整から始めなければならず、SLO は組織のルールになりえない。

## それでも SLO が難しいのは、誰もが壊したくないのに、引き受ける損失の種類が違うからである

SLO の議論を雑にすると、「開発は速く出したい、運用は壊したくない、ビジネスは儲けたい」といった単純な対立図式になりがちである。しかし、これは説得力の弱い見方である。開発側だって壊したくない。障害が起きれば自分たちが呼ばれ、信用を失い、次の開発も遅れる。運用側だって、ただ止めたいわけではない。ビジネスや開発の速度が落ちることは理解している。ビジネス側だって品質を軽視しているわけではない。問題は、誰もが壊したくないと思っているにもかかわらず、それぞれが異なる損失により敏感だという点にある。

開発側は、変更を出せないことによる損失、すなわち価値提供の停滞やリードタイムの悪化に敏感である。運用側は、障害時に集中して発生する負荷、すなわちオンコール、復旧、問い合わせ、再発防止対応といったコストに敏感である。ビジネス側は、ユーザー損失や機会損失に敏感である。つまり、対立しているのは善悪ではなく、損失関数の違いである。

この理解に立つと、SLO の役割は「無責任な開発を止めること」ではなく、「全員が壊したくないと思っているにもかかわらず、どの程度のリスクをどの条件で受け入れるかについて感覚がずれやすい状況に、共通の境界を与えること」だとわかる。SLO は速度と安定性のどちらか一方の味方ではない。むしろ、どの状態までは速度を優先し、どの状態からは安定性を優先するかを固定するためのルールである。

## では、どうやってそのルールを握るのか

ここで核心に入る。SLO を握るとは、正しい数値を見つけることではない。守るべき体験、観測可能な量、境界、行動、コスト負担の対応関係を、組織の共通ルールとして固定することである。そのためには、少なくとも三つの段階を踏まなければならない。

第一に、出発点を数値ではなく「守る体験」に置く必要がある。たとえば注文 API の議論をするなら、「成功率を何パーセントにするか」から始めてはならない。まず「注文完了が崩れると何が起きるか」から始めるべきである。注文が失敗すれば、ユーザーは目的を達成できず、再試行が発生し、問い合わせも増え、売上機会も失われる。ここでようやく、技術の議論がビジネスや開発の意思決定に接続される。もし最初から「p95 が 400ms です」とだけ言えば、その数値が何の体験崩壊を意味しているのかが見えず、説得材料になりにくい。

第二に、その守る体験を観測可能な量に翻訳しなければならない。ここで SLI の選定が入る。注文完了を守りたいなら、注文成功率や注文処理タイムアウト率が候補になる。一覧表示の快適さを守りたいなら、一覧 API のレイテンシやリトライ率が候補になる。このとき重要なのは、「測りやすいから選ぶ」のではなく、「守るべき体験に最も近いから選ぶ」ことである。なぜなら組織と握る際に必要なのは、「この数値の悪化は、先ほど合意したこの体験の崩れを意味する」と説明できることだからである。

第三に、その SLI に対して、境界と行動をセットで提示しなければならない。ここが最も重要である。たとえば「注文成功率を主要 SLI とし、99.5% を下回ったら新規リリースを停止して原因調査に入る。99.0% を下回ったらインシデントとして扱う」といった形である。この提示の仕方によって初めて、議論は数値の好みから、どの損失をどこまで引き受けるかという実質的な論点に移る。

当然ながら、ここでは衝突が起きる。開発側は「その程度でリリースが止まるのは重い」と言うかもしれない。ビジネス側は「そこまで止める価値が本当にあるのか」と問うかもしれない。しかし、この衝突はむしろ必要である。そこで初めて、どの損失をどこまで許容するかという実質的判断が表面化するからである。SLO を握るとは、この衝突を避けることではなく、衝突を可視化したうえで、調整可能な形に変えることである。

## 調整は、数値だけでなく、行動の重さと適用範囲も含めて行う

ここで重要なのは、交渉の対象が閾値だけではないということである。SLO をめぐる議論は、しばしば「99.5% にするか 99.9% にするか」といった数値の調整に矮小化される。しかし実際には、調整すべき変数は少なくとも三つある。閾値そのもの、閾値を越えたときの行動の重さ、そしてそのルールをどこに適用するかである。

閾値が厳しすぎるなら、少し緩めることができる。だがそれだけが手段ではない。たとえば 99.5% を下回ったときの行動を、即時リリース停止ではなく、まず追加レビュー必須や調査優先に落とすこともできる。また、すべての API に同じ厳しさを適用するのではなく、注文、認証、決済のような重要導線だけに厳格に適用することもできる。つまり、握るとは「一番きれいな数値を選ぶこと」ではなく、「組織が実際に従えるルールになるように、閾値・行動・適用範囲の三つを釣り合わせること」である。

ここで使う説得材料は、少なくとも三種類ある。第一に、過去事例である。以前にこの API の障害が問い合わせ増加や売上影響につながったという事実があれば、それは非常に強い。第二に、現実の品質分布である。平常時にはどの程度で、障害時にはどこまで落ちるのかを見せることで、提案している境界が恣意的ではなく、運用上意味のある線であることを示せる。第三に、行動コストである。この境界を越えたら何が止まり、どれくらい遅れ、誰が何を負担するのかを明示することで、議論は抽象論ではなく実務的な意思決定になる。

エラーバジェットは、この握りを持続可能にする

単発の閾値違反だけを見ていると、議論はどうしても瞬間的になりやすい。ノイズで過剰反応したり、逆に慢性的な劣化を見逃したりしやすい。ここでエラーバジェットが効いてくる。エラーバジェットは、「今どれだけ悪いか」だけではなく、「一定期間の中で、どれだけ失敗の余地をすでに使ったか」を示す。これによって議論は時間軸を持つ。

この視点に立つと、開発側に対しては「ここまでは進めるが、ここから先は止める」という説明がしやすくなる。ビジネス側に対しては「必要以上に過保護な運用をしたいのではなく、許容されたリスクを超えたら優先順位を変える」という説明がしやすくなる。つまりエラーバジェットは、SLO を単なる禁止や抑制ではなく、攻めと守りを調整する共通ルールに変える。

## むすび

SLI、SLO、SLA、エラーバジェットを理解するうえで重要なのは、それらを静的な管理用語として見ることではない。SLI は守るべき体験を観測可能な量に翻訳するための概念であり、SLO はその体験がどこまで崩れたら組織の優先順位を切り替えるかを決める境界であり、SLA はその一部を外部契約にしたものであり、エラーバジェットはその境界を開発速度と接続するための概念である。

したがって、SLO の良し悪しは、理論的にもっともらしいかどうかで決まるのではない。その境界を越えたときに、誰が、何を、どのコストを払って変えるのかが事前に合意されているかどうかで決まる。SLO を握るとは、まさにその合意を作ることにほかならない。
